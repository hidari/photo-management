# Relay プロジェクト 議論まとめ

## 1. ローカルSQLite + S3互換ストレージによる同期の現実性

そもそもの問いとして「モバイルアプリでローカルSQLiteを使いつつ、S3互換ストレージを中央の保管庫として他ユーザーとデータをやりとりできるか」という議論から始まった。

結論としては「設計次第では現実的だが、素朴にやると破綻する」というもの。S3はオブジェクトストレージであってデータベースではないため、複数ユーザーが同時に書き込んだ場合のコンフリクト解決の仕組みが一切ない。Last-Write-Winsで黙って上書きされる。

対処策として、変更差分をイベントログ（チェンジセット）として個別のS3オブジェクトに書き出してクライアントが適用する方式（CRDTの考え方）や、cr-sqliteのようなSQLiteのCRDT拡張、あるいはTurso・PowerSync・Electric SQLといった専用の同期エンジンを挟む方法がある。


## 2. Tursoとは何か

Tursoは「SQLiteをクラウドで動かせるようにしたマネージドDBサービス」。SQLiteをフォークしたlibSQLというOSSを土台にしており、SQLiteにHTTP APIサポートやレプリケーション機能を追加している。

最大の特徴は「Embedded Replica（組み込みレプリカ）」で、クラウド上のプライマリDBをローカルのSQLiteファイルとして端末に持てる。読み取りはローカルから一瞬で返り、書き込みはクラウドのプライマリに飛んで差分が非同期でローカルに反映される。つまりローカルファーストな動作とクラウド同期を両立できる。

なお現在TursoはlibSQLフォーク路線と並行して、SQLiteをRustで一から書き直したTurso Databaseエンジンの開発に注力している。こちらはネイティブasync I/O・ベクター検索・並行書き込みを目標としているが、まだプロダクションレディではない。

### Tursoの料金

無料プランはクレジットカード不要で5GBストレージ・月5億行読み取り・月1000万行書き込みまで使える。Developerプランは月$4.99（年払い）で月25億行読み取りまで拡張される。有料プランであればデータベース数は無制限。


## 3. RelayアーキテクチャとTursoの関係

Relayの設計を読んだ結果、現在のPhase 1〜2においてはTursoは不要という結論になった。

Relayの設計の本質は「データの流れが常に一方向であること」にある。カメラマン → Google Drive → モデルという流れで、Relayのサーバーは何も永続化しない。カメラマンとモデルの間でやりとりされるのは写真ではなくGoogle DriveのURLだけであり、モデル側のアプリはそのURLからファイルリストを表示するだけでよい。

この設計においてTursoを使おうとすると、「ストレージはユーザーのクラウドに任せてRelayはサーバーを持たない」という設計の清潔さが崩れてしまう。ただし、カメラマンが複数デバイス間でメタデータを同期したい場合には将来的に選択肢として浮上する。


## 4. リレー機能の追加とアーキテクチャへの影響

「モデル間でのデータの受け渡し」や「集合写真の加工リレー」機能を追加する場合、アーキテクチャに本質的な変化が生じる。

現在の設計が成立している理由はデータの流れが一方向だからだが、加工リレーでは「誰が今この写真を持っているか」「誰がまだ返送していないか」という状態を複数ユーザー間で共有する必要が生まれる。これは本質的に協調（coordination）の問題であり、中央に状態を持つ場所が必要になる。

ここで初めてTursoが有力な選択肢になる。ワークフローの状態はSQLiteで十分表現できるデータ量で、ファイルの実体は引き続きユーザーのクラウドストレージに置いたまま「誰が何の状態にいるか」だけを管理する薄いバックエンドを立てる構成が、設計の清潔さをなるべく保ちながら協調を実現する現実的な落としどころとなる。

加工リレーのデータモデルとしては、シーケンシャルな「リレー（駅伝型）」と全員が並行して加工する「並列収集（アンケート回収型）」を別物として設計しておくことが重要。

```
EditSession {
    id: UUID,
    source_photo_id: String,       // 元写真
    participants: Vec<Participant>, // 参加者リスト
    current_step: usize,           // 現在誰の番か
    status: EditStatus,            // Pending / InProgress / Complete
}

EditStep {
    session_id: UUID,
    participant_id: String,
    received_at: DateTime,
    submitted_at: Option<DateTime>,
    result_file_url: Option<String>, // 加工済みファイルのURL
}
```


## 5. ビジネスモデルの整理

### 価格設定

- 配布機能（カメラマン・モデル共通）: 買い切り500円
- リレー機能: 月300〜500円のサブスクリプション

この構成はフリーミアムの教科書通りの構造で、「配布の受け取り」という無料体験（実際は500円の買い切りだが心理的な初回接触として機能）でサービスを知ってもらい、より高度な機能に課金するという自然な導線を作っている。

### 「モデルはお金を出さない」という前提の正確な意味

モデルが全くお金を払わないわけではなく、財布の紐が固い層という意味。そのため「まず500円を払ってでも欲しい体験」を配布機能で提供し、リレー機能では「これがないと困る」「これがあると嬉しい」がはっきり見える価値を示すことが課金の前提条件になる。

### コスト構造の強み

Relayのストレージをユーザー自身のGoogle Driveに乗っかる設計にすることで、一般的なファイル共有サービスのコストの大半を占めるストレージとCDN（配信帯域）費用をゼロにできる。これは単なる実装上の工夫ではなく、「サービスが維持できる程度+α」という価格帯で持続可能なビジネスを作るための根本的な設計判断として機能している。

Phase 1〜2の配布機能だけを見れば、継続的なインフラコストはほぼApple Developer Program年会費（$99）とGoogle Play Developer登録料（$25 one-time）のみ。

### App Storeの手数料

サブスクリプションをApp Store経由で販売する場合、Appleの手数料30%（小規模デベロッパー向けには15%）が引かれる。300円のサブスクから手元に残るのは210円程度になる計算で、インフラコストとの兼ね合いを初期から意識しておく必要がある。


## 6. UXの重要性

技術的な差別化と同等かそれ以上に、UXが500円の価値判断に直結する。ギガファイル便の問題は広告だけでなく、「ファイル転送ツールとしての体験が雑」という根本的な課題にある。

特に重要な観点として以下が挙がった。

わかりやすさとサクサク感については、待ち時間の実際の長さよりも「何が起きているかわからない待ち時間」の方がストレスとして大きく感じられる。プログレス表示やスケルトンUIなど「ちゃんと動いてますよ」を伝える仕組みを最初から丁寧に実装する価値がある。

サムネイルの表示については、「全部揃ってから表示」ではなく「取れたものから順に表示」にするだけで体感速度が大きく変わる。

通知については、XなどのSNS系でリアルタイム通知に慣れたユーザーには通知がないとUXとして致命的になりうる。カメラマンがリンクを送付した際にモデルがプッシュ通知を受け取れる体験は必須級の機能。

初回起動体験については、リンクを受け取ったモデルがアプリを開いた瞬間に「これは信頼できるやつだ」と感じてもらえるかどうかが500円の価値判断に直結する。Phase 2のモバイルアプリではオンボーディングフローに時間をかける価値がある。


## 7. 通知インフラ（FCM + Cloudflare Workers）

### FCMのコスト

Firebase Cloud Messaging（FCM）自体は送信数に関わらず完全無料。ただしFCMを叩くサーバーサイドのコード実行コストは別途考慮が必要。

### Cloudflare Workersとの組み合わせ

Cloudflare Workersの無料プランは1日10万リクエスト（月換算約300万リクエスト）。Relayのユースケースで「1イベントあたり平均3人のモデルに通知する」と仮定すると、月100万イベントをこなして初めて上限に近づく計算になり、国内コスプレ・ポートレート市場の規模では当面無料枠に収まる。

有料プランに移行した場合も追加100万リクエストあたり$0.30程度なので、月1000万通知でも追加コストは$21ほど。この規模のユーザーベースがあればサブスク収益でとっくに賄える。

### 心配すべき順番

通知コストよりも、リレー機能のワークフロー状態管理（TursoのSQLite）の方が先にコスト問題になる可能性が高い。ただそちらも月$5程度で相当な規模まで対応できる。FCMのコストを心配すべきタイミングは、Relayが予想外に国際的に広がるような嬉しい誤算が起きたときくらい。


## 8. 開発の進め方の方針

以下の順番でリスクを小さく積み上げていくことが合理的という結論になった。

Phase 1〜2（配布機能）を先に完成させる理由は二つある。一つはリレー機能がサーバーサイドの協調ロジックを必要とするため、その土台となるデータモデルや認証フローを配布機能で固めてからの方が設計の無駄が少ない。もう一つは、実際のカメラマンに配布機能を使ってもらうことで「リレー機能で本当に困っているか、どういうフローで使いたいか」という具体的なフィードバックが得られることで、現時点のリレー機能はまだ合理的な仮説の段階にある。

relay-coreのトレイト設計だけは「将来リレー機能が乗っても崩れない」レベルを意識しておくことが推奨される。具体的にはNotificationProviderの抽象化はすでに良い設計になっているが、Distributionのデータモデルにもリレーセッションと紐付く可能性を考慮した拡張ポイントを小さく残しておくイメージ（フィールドを一個予約しておく程度で十分）。
